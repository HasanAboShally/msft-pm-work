# Fabric Local MCP ‚Äì Latest Project Status and Updates (Dec 2025)

**The Fabric local MCP server is currently in public preview**, with the first beta releases now available and ongoing enhancements underway. This open-source **MCP** (Model Context Protocol) server provides AI agents with comprehensive context for Microsoft Fabric‚Äôs public APIs, item definitions, and best practices, all running locally for security. Below we summarize where the project stands today ‚Äì including recent developments, current status, and upcoming plans ‚Äì with key dates and contributors: [\[blog.fabri...rosoft.com\]](https://blog.fabric.microsoft.com/en-us/blog/introducing-fabric-mcp-public-preview/)



Public Preview Status
The Microsoft Fabric MCP server is in Public Preview (beta) as a local, offline tool that packages Fabric‚Äôs entire REST API schema and item definitions for AI-assisted development. It was open-sourced in October 2025 and is part of the broader Microsoft MCP initiative, but remains preview-only while Azure‚Äôs MCP has reached GA.

First Beta Releases
The team delivered its first beta releases in November 2025. Version 0.0.0-beta.1 debuted mid-November, followed by 0.0.0-beta.2 a week later. These pre-release builds are published across multiple channels ‚Äì including a Visual Studio Code extension (‚ÄúMicrosoft Fabric MCP Server‚Äù), a NuGet package (`Microsoft.Fabric.Mcp`), and an NPM package (`@microsoft/fabric-mcp`) ‚Äì enabling internal and external users to easily install and experiment with Fabric MCP.

Ongoing Enhancements
Active development is adding new capabilities. A major recent addition is an ‚ÄúInvoke API‚Äù mode that lets the server expose a single dynamic tool for any REST endpoint (besides the classic per-API tools), introduced to compare approaches for clients. The team is also integrating contributions from other Fabric groups ‚Äì for example, the OneLake team is adding file management commands (upload/download/list for OneLake storage) into the Fabric MCP server ‚Äì expanding its scope. These enhancements aim to enrich the MCP‚Äôs functionality as it marches toward General Availability.

## Current Status and Features

**As of today, Fabric MCP remains in *public preview*** ‚Äì not yet GA ‚Äì with version **0.0.0-beta.2** being the latest. It runs as a local service that **does not execute actions in the Fabric tenant**; instead it provides the full OpenAPI specs and JSON schemas of Fabric‚Äôs public APIs to assist code generation. All Fabric workloads (Lakehouses, Pipelines, Real-Time Analytics, etc.) are represented, allowing AI copilots to retrieve operation details, required parameters, and sample structures. The server adheres to a **‚Äúlocal-first‚Äù security model**, meaning it holds all context offline and never directly calls the cloud APIs ‚Äì the human developer remains in control to run any generated code. Being open-source and part of the Microsoft MCP repository, Fabric MCP invites community contributions and aligns with a growing ecosystem of MCP servers for various services (Azure DevOps, SQL, etc.). [\[blog.fabri...rosoft.com\]](https://blog.fabric.microsoft.com/en-us/blog/introducing-fabric-mcp-public-preview/) [\[blog.fabri...rosoft.com\]](https://blog.fabric.microsoft.com/en-us/blog/introducing-fabric-mcp-public-preview/), [\[blog.fabri...rosoft.com\]](https://blog.fabric.microsoft.com/en-us/blog/introducing-fabric-mcp-public-preview/)

**Availability:** Developers can access Fabric MCP in multiple forms. The team has published it as a **VS Code extension** (titled *‚ÄúMicrosoft Fabric MCP Server‚Äù* in the marketplace) for easy launch in the IDE, as well as a **NuGet package** (`Microsoft.Fabric.Mcp`) and an **NPM package** (`@microsoft/fabric-mcp`) for those who prefer to run it via .NET or Node.js tooling. (A Docker image is not yet provided ‚Äì currently **‚ÄúN/A‚Äù** ‚Äì but can be added on demand if needed. In fact, user feedback has already requested a container option.) With these distribution channels in place, internal users and preview customers can experiment with Fabric MCP easily: for example, connecting GitHub Copilot or other AI agents to the local server to generate Fabric automation scripts. **Functionality:** The Fabric MCP server exposes **tools corresponding to Fabric REST API endpoints and object definitions**. By default (‚ÄúClassic‚Äù mode), it offers one tool per API (e.g. separate tools for creating a Lakehouse, listing workspaces, etc., mirroring each public REST operation). In late September, an **additional mode (‚ÄúInvokeApi‚Äù)** was introduced, which instead exposes a single generic `Invoke API` tool that can call any endpoint by path and method. There is also a **‚ÄúHybrid‚Äù mode** combining both sets of tools. These options were added to **evaluate which approach better serves users** (multiple specific tools vs. one flexible tool). The feature was implemented by Mahir Diab and merged on Sep 28, 2025, expanding the server‚Äôs versatility. Other built-in capabilities documented include providing **JSON schemas for each Fabric item type** (so an AI can know the structure of a Dataset, KQL database, Warehouse, etc.), and embedding Microsoft‚Äôs **best-practice guidance** (pagination patterns, error handling advice, common pitfalls) within the context returned to the AI agent. This means the code suggestions you get from an AI (like Copilot) using Fabric MCP should automatically follow Fabric‚Äôs recommended patterns (for example, handling continuation tokens on list queries, or waiting on long-running operations properly). All of this is packaged in the offline server, ensuring **agents have rich knowledge without needing direct access to the Fabric service**. ## Internal Developments and What‚Äôs Next

Internally, the Fabric MCP project has been a cross-team effort, and recent communications shed light on immediate next steps:

*   **Finalizing Preview Deliverables:** By late November, the focus was on getting all release artifacts out. Collaboration with the Azure MCP engineering team (who built the shared release pipeline) was key to publishing the VSCode extension under the official Fabric publisher identity. With that done, the **Public Preview is feature-complete in terms of packaging**. Documentation has been another focus ‚Äì comprehensive docs (README, changelog, troubleshooting guides) for Fabric MCP are being added to the repo to help external adopters. *   **Integration of New Commands (OneLake & More):** As noted, the OneLake team is actively contributing new **storage operation tools** to Fabric MCP. In an Oct 29 update, Amos Hersch shared that OneLake engineers want to add commands for file uploads, downloads, directory listing, etc., specifically targeting Fabric‚Äôs OneLake storage API. This effectively would let the Fabric MCP not only *generate code* (as it does now) but possibly *execute file operations* in a controlled way. The Fabric team is evaluating how to incorporate this: ensuring the number of tools remains reasonable (possibly grouping these under a category) and that **authentication is handled securely** (likely leveraging patterns from the Azure MCP server). Since current Fabric MCP tools don‚Äôt require user credentials (they just produce code), adding action-oriented tools means addressing auth and security review for those parts. The plan is for the OneLake contributors to handle the security review of their additions, under Fabric team guidance. This effort signals a broader **beyond-read-only expansion** ‚Äì allowing certain **execution or management tasks** via MCP, which blurs the original line between ‚Äúlocal codegen MCP‚Äù and ‚Äúremote execution MCP.‚Äù Indeed, in an internal discussion Yifat Sagiv noted that *‚Äúour separation that local MCP is about code generation, and remote is about execution doesn‚Äôt necessarily hold‚Äù*, as even some ‚Äúlocal‚Äù MCP servers (like Power BI‚Äôs) might implement execution commands. The Fabric team is navigating this carefully, balancing the preview‚Äôs scope with valuable contributions. We can expect support for OneLake operations to land in upcoming releases, given the strong motivation and active work through November. *   **Alignment with Azure MCP (Unified MCP initiative):** The Fabric MCP is one part of the broader **‚ÄúMicrosoft MCP‚Äù** ecosystem, which includes Azure services‚Äô MCP servers. There have been conversations about how Fabric‚Äôs server will align with Azure‚Äôs going forward. Notably, in August a plan was mentioned from Azure leadership (Josh Free) about **‚Äúone big local MCP‚Äù** that would include Fabric and other services in a unified experience. This suggests a future direction where users might not run separate MCP servers per service, but have a single combined server or a more integrated setup. For now, however, Fabric MCP is **developed and released separately**. At Microsoft Ignite 2025 (mid-November), the Azure MCP server reached **General Availability (1.0)** with its own announcements and demos, but those did **not yet include the Fabric MCP**. In an internal thread, PM Aviv Ezrachi asked if Fabric MCP was GA ‚Äúas part of the Microsoft MCP‚Äù at Ignite, to which Amos clarified: *Azure‚Äôs MCP server is GA, but that release ‚Äúdoesn‚Äôt include our MCP server‚Äù* ‚Äì Fabric MCP stays in preview for the time being. Going forward, as Fabric MCP matures, we anticipate closer coordination with the Azure MCP team. The infrastructure (build/release pipeline) is already shared to some extent, and once Fabric MCP proves out in preview, a path to **graduate it to GA** (possibly version 1.0 aligned with the unified MCP product) will be planned. This may involve synchronizing versions, ensuring parity in features (e.g. Azure‚Äôs MCP recently added GA polish like managed identity support, which Fabric MCP might adopt), and co-announcing its General Availability at a future date. **In summary,** the Fabric local MCP project has quickly progressed from concept to public preview and first releases. The latest updates have solidified its distribution (VS Code extension, etc.) and added flexible invocation capabilities. For now, **Fabric MCP‚Äôs preview is live and thriving**, with enthusiastic internal support and a clear trajectory to become a key piece of the Fabric developer experience.
