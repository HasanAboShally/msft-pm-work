# **Preparing Fabric Local MCP for Import/Export of Item Definitions**

**Microsoft Fabric‚Äôs Local Managed Control Plane (MCP)** is an AI-focused framework that runs on a developer‚Äôs machine to provide **rich context (API specs, item schemas, best practices)** for AI agents. It keeps the user in control by generating code rather than executing it directly. As we look to enhance Fabric MCP with **import/export of item definitions** (for reports, datasets, pipelines, etc.), it‚Äôs useful to examine how others handle similar tasks, and what UX patterns support a human-in-the-loop AI agent model. Below is a comprehensive breakdown of current practices, tools, and design considerations: [\[blog.fabri...rosoft.com\]](https://blog.fabric.microsoft.com/en-us/blog/introducing-fabric-mcp-public-preview/)


## Brief 

### üí° Evolving MCP: From Read-Only Context to Action
Today, Fabric‚Äôs MCP provides read-only context ‚Äì schemas and API details ‚Äì so AI can draft code for creating or updating Fabric items. Upcoming enhancements aim to let agents execute actions as well, starting with local tasks (e.g. uploading files) and eventually leveraging Fabric‚Äôs own APIs for creating items. This evolution will bridge the gap between code generation and direct automation, guided by human oversight.

### üìÅ Import/Export APIs Simplify Deployment
Manually cloning or migrating Fabric content is complex. Internal experts currently script the process: retrieve each item‚Äôs definition, then create items in the correct order while fixing IDs and connections. A new batch Import/Export API (previewing early 2026) will export an entire workspace‚Äôs item definitions in one go and import them intact, auto-preserving relationships (no manual ID fixing). Fabric MCP can build on this capability to support one-step workspace duplication workflows.


## **1. How Others Use Local MCP Servers for Item Definitions**

**Internal Use Cases ‚Äì** *Accelerating deployment & cloning:* Before dedicated APIs, teams at Microsoft have managed content deployment by treating item definitions as portable templates. Items like Lakehouses must be created first to obtain new IDs, which then get substituted into downstream item definitions (notebooks, pipelines, semantic models) before creation. Connections and binding (e.g. linking a pipeline to a Lakehouse, or a dataset to a storage account) also require adjustment in the definitions, making the process intricate. This code-first approach has been used in CI/CD automation (with internal libraries like **fabric-cicd** and custom scripts) to propagate changes across multiple workspaces. It treats definitions as source code ‚Äì checked into repos and applied via scripts or pipelines. While powerful, it demanded careful ordering and ‚ÄúID patching‚Äù logic to maintain relationships. ## **2. Best Practices & Inspirations for Import/Export Workflows**

Whether in Fabric or analogous platforms, a few best practices emerge for definition import/export:

*   **Work in Bulk for Consistency:** Migrating content item-by-item is error-prone. The best practice is to **export entire configurations in bulk** (all related items together) and import them together. This ensures referential integrity. Fabric‚Äôs new batch API embodies this by exporting a whole workspace in one call. Similarly, other platforms like **Power BI** have moved toward project files (PBIP) containing an entire report‚Äôs assets. By handling multiple objects in one package, you avoid version mismatches and missing dependencies. *   **Maintain Relationships Automatically:** A major pain point is re-linking items after import (for instance, connecting a cloned dataset to the new copy of a lakehouse). In other words, let the platform handle identity mapping. This is analogous to how **infrastructure-as-code tools** (like ARM or Terraform) let you define resources by name and automatically resolve the actual IDs on deployment. Indeed, Microsoft‚Äôs guidance is that the new API will spare developers from manual GUID juggling. ## **3. UX Patterns for AI Agent‚ÄìDriven, Human-Guided Workflows**

Designing Fabric MCP features for a world where **AI agents act, but humans supervise**, requires careful UX considerations. The goal is to harness the speed and scale of AI automation without sacrificing the user‚Äôs sense of control or clarity. Some key patterns and ideas:

*   **‚ÄúReview and Approve‚Äù Loop:** This is fundamental to Fabric MCP‚Äôs philosophy. In the initial release, the MCP only **suggests actions (by providing code or commands)** and leaves execution to the user. Users see, for example, a Python script that the agent generated to create a Lakehouse, and they consciously run it. As we add direct execution capabilities, we should preserve an explicit confirmation step for major changes. An example to emulate is Azure MCP‚Äôs *elicitation support*, which **prompts the user for confirmation before an agent executes any tool that might return sensitive data or make a significant change**. In our context, an AI attempt to import a batch of items to a workspace could trigger a confirmation: ‚ÄúThe AI is about to create 5 new items in workspace X. Proceed?‚Äù This ensures the human operator stays in the decision loop. It‚Äôs a safety check that fits the ‚Äúhuman-in-the-loop‚Äù model. *   **Transparency of Actions:** When an AI agent performs steps on the user‚Äôs behalf, the user interface should show those steps clearly ‚Äì almost like a playback of what‚Äôs happening. One approach is to log all actions in a **console or output panel** (e.g., listing each REST API call or CLI command executed, and the result). This is analogous to how the Fabric CLI prints what it‚Äôs doing, or how deployment pipelines list each task. It builds trust: the user can see ‚ÄúImported dataset ABC ‚Äì Success‚Äù or any errors with context. Some IDE integrations with MCP already do this; for instance, VS Code‚Äôs Agent Mode orchestrator will show the conversation and tool use. We might extend that with a **rich diff or summary** after an import: e.g., a report of ‚Äú5 items created, 2 updated, 0 errors.‚Äù

*   **Inline Editing and Corrections:** Given that AI might not always get everything perfect, allowing the human to intervene and tweak is important. For example, if an agent drafts a CLI script to import definitions but the user spots a mistake (perhaps the wrong workspace ID), the UI should let them edit the script before running. This pattern is akin to how GitHub Copilot might suggest code which you then edit. In a Fabric context, we might present a generated PowerShell or Fabric CLI script for review; the user can modify any parameters or commands, then execute it. This merges AI assistance with human expertise gracefully. In tooling terms, this could be a simple text editor UI in the Fabric portal or VS Code where the generated script appears with an ‚ÄúEdit and Run‚Äù option.

*   **Tool ‚ÄúGenerators‚Äù vs ‚ÄúExecutors‚Äù:** One UX approach is to separate the concept of generating code from executing it. For instance, we could have an MCP **tool that generates a CLI script** for a given task (without running it), and another tool that runs a provided script. The agent, under the hood, first calls the generator (producing a script as output), then asks the user for approval, then calls the executor with that script. This two-step pattern was hinted at by internal discussions: *‚ÄúTriggering a CLI script can be a great addition as a pre/post action‚Äù* in deployments. It suggests the agent can produce a script to do something and then that script could be triggered at the right time. By codifying this in the MCP design, we make the process safer and more modular. The user can even save the script for reuse. # **Fabric Local MCP ‚Äì Upcoming Enhancements (Focus on Item Definition Workflows)**

**Objective:** Enable AI agents (with human oversight) to **manage Fabric item definitions** (reports, datasets, pipelines, etc.) seamlessly. This includes robust **import/export** of definitions, safe execution of actions, and new tools to streamline definition-based workflows.

## **Core Planned Features & Requirements**

*   **Generalized Definition Support:** Expand Local MCP to handle *all item types‚Äô definitions* (e.g. reports, datasets, pipelines) in a unified way. Initially focus on reports, then broaden to other items. *   **Import/Export API Integration:** Leverage the new **batch Import/Export APIs** (coming early 2026) to move entire workspace definitions in one go. This will preserve inter-item relationships automatically (no manual ID fixes).
    *   *Benefit:* One-step cloning or backup of a workspace by exporting all item JSON definitions and re-importing them elsewhere. Simplifies what used to require custom scripts for each item.

## **Proposed New Features (Inspired by Research & Feedback)**

*   **CLI Script Generation Tool:** Add an MCP tool (e.g., `GenerateCLIScript`) that produces Fabric CLI or PowerShell scripts for a given high-level task. For instance, the agent could output a script to **export a workspace and import it to another** (using the Fabric CLI commands for batch export/import). This gives the user a concrete artifact ‚Äì they can run it as-is or modify it. It aligns with the idea of giving users a tangible asset for complex actions and leverages the Fabric CLI‚Äôs robust capabilities.     *   *Use Case:* Agent generates a script with steps to create a set of new items (Lakehouse, then Dataset, then Report) in the correct order. The user can review that script in their IDE or terminal and execute when ready.
*   **CLI Execution Tool:** Complement script generation with a `RunCLICommand` tool in MCP. After a script is reviewed, the user (or agent with permission) can invoke this tool to execute the CLI commands. This effectively lets the AI **carry out multi-step operations** in Fabric through the trusted CLI interface, all under the hood of MCP. It ensures consistent authentication and error handling via CLI‚Äôs mechanisms.
    *   *Safety:* The execution tool will always either require a confirmation or return the output for verification. This keeps the user in control while offloading the tedious execution to the agent.
*   **Definition Diff & Validation Utilities:** Provide a way to compare item definitions or validate them. For example, a `CompareDefinition` tool could show differences between a dataset‚Äôs current definition and a modified one (highlighting what an agent changed). A `ValidateDefinition` tool can check if a JSON definition meets the schema and report errors before trying an import. These features help catch mistakes early and support collaborative human-AI editing of definitions.
*   **Enhanced UX for Agents in IDE/Portal:** Ensure that working with definitions via AI feels intuitive:
    *   In VS Code, when an agent retrieves a definition (say a Pipeline JSON), format it nicely and perhaps split into sections for easy reading. If the agent suggests changes, present them as a **diff or pull request-style commentary**, so the developer can accept or reject specific changes.
    *   In the Fabric web portal (for pro users): consider an **embedded CLI console or ‚ÄúCopilot panel‚Äù** where agent-generated commands can be displayed and run. This would cater to power users who want to see CLI actions live. Alternatively, a simple *‚ÄúApply Changes‚Äù* button on an agent‚Äôs suggestion card, with a behind-the-scenes call to our execution tool, could streamline the process for less technical users.

    1.  Agent calls an Export tool ‚Äì gets a bundle of definitions.
    2.  Agent optionally asks user for any target specifics (new names, etc.).
    3.  Agent calls an Import tool or generates a CLI script for import.

## **Slide-Friendly Summary (Key Points)**

*   **Empowering Definition-Centric Workflows:** Fabric Local MCP will support *all item definitions* (reports, datasets, pipelines, etc.) as first-class objects, enabling easy retrieval, editing, and deployment of these definitions by AI agents (with oversight).
*   **One-Click Import/Export:** **New batch APIs** for exporting/importing entire workspaces will be integrated. Agents can help clone or back up a workspace in one shot, preserving all links between items. (No more manual JSON stitching!)
*   **AI + Human Collaboration:** The design emphasizes a **hybrid approach** ‚Äì AI prepares the work (e.g., generates code or scripts), humans review and approve execution. This fosters trust and aligns with customer feedback that complex tasks should yield visible outputs, not be a black-box. We‚Äôre addressing those gaps (providing official, supported ways to manage definitions) and borrowing good ideas (like **script libraries** and **agent orchestration of multi-step tasks**) to make Fabric‚Äôs MCP the best-in-class tool for AI-assisted development.
