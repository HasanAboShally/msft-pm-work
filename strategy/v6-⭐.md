## I. THE WORLD IS CHANGING

### Platforms Are Externalizing Capabilities

Fabric is powerful inside the portal, but increasingly work happens outside it â€” in CI/CD pipelines, automation scripts, and AI agents. Platforms are externalizing their capabilities to meet users where they are.

**"Embedding Fabric Everywhere"** means exposing Fabric's full capabilities programmatically, so developers can automate and AI agents can operate. This positions Fabric as infrastructure you call, not just a portal you visit â€” for example, building a data pipeline from ChatGPT or managing deployments from VS Code.

We propose building on our team's strengths â€” automation, embedding, platform ownership, and MCP infrastructure â€” to externalize Fabric capabilities and meet users where they are.

### The End of Click-Ops

A fundamental shift is underway in how people interact with platforms.

Instead of navigating portals, clicking through menus, and manually configuring settings, users increasingly expect to **state their intent** and have systems figure out the rest. The evidence is everywhere:

- **Search â†’ Chat**: Users are replacing traditional web searches with conversational AI. A 2024 SparkToro/Datos study found that **58.5% of US Google searches now end in zero clicks** â€” only 360 out of every 1,000 searches reach the open web [1]. Gartner predicts a **25% decrease in traditional search engine volume by 2026** as AI chatbots and agents absorb queries [2]. Even within search engines, AI-generated summaries are displacing link-clicking behavior.
- **Websites â†’ On-the-fly UI**: The MCP protocol â€” already adopted by VS Code, Claude, ChatGPT, and thousands of tools â€” now supports [MCP Apps](https://modelcontextprotocol.io): interactive, embeddable UI components served by any tool provider, rendered inline wherever the user is. In just nine months since launch, the Postman 2025 State of the API Report shows **70% of developers are already aware of MCP**, with 10% using it regularly and 24% planning to explore it [3].
- **Portals â†’ Embedded Experiences**: Rather than switching to a platform's portal, users expect the platform to come to *them* â€” integrated into their IDE, their chat, their workflow.

> **The era of classic portals is at risk.** Platforms that don't externalize their capabilities â€” both programmatically and experientially â€” risk becoming invisible in an AI-mediated world.

### What This Means for Fabric

Microsoft Fabric is a **powerful SaaS platform** â€” and that has brought real strengths: simplicity, unified experience, and accessibility for non-technical users. But today, **not all Fabric capabilities are accessible outside the portal**. To stay relevant, Fabric needs to become fully embeddable and automatable â€” meeting users wherever they work.

Meanwhile, the world outside is accelerating:

- **76% of developers** use or plan to use AI coding assistants, up from 70% the prior year â€” with **62% actively using** them today ([Stack Overflow Developer Survey 2024](https://survey.stackoverflow.co/2024/ai)) [4]. Among API professionals, the figure is even higher: **89% use generative AI in their daily work** ([Postman State of the API 2025](https://www.postman.com/state-of-api/)) [3]
- **MCP** has been donated to the Linux Foundation (now under LF Projects, LLC), becoming the "HTTP for AI agents" [5]
- Autonomous agents are expected to handle an increasing share of platform operations, from provisioning to troubleshooting. The AI agents market is projected to reach **$52.6 billion by 2030** at a 46.3% CAGR ([MarketsandMarkets, 2025](https://www.marketsandmarkets.com/Market-Reports/ai-agents-market-15761548.html)) [6]
- Today's API throttling limits (designed for human-scale usage) will face **10â€“100x pressure** as millions of agents begin interacting with platforms. Already, **51% of developers cite unauthorized or excessive agent API calls** as their top security concern [3]

**If Fabric doesn't adapt, it risks being trapped in a portal-only paradigm while users â€” and their agents â€” move on.**

## II. OUR THESIS: EMBEDDING FABRIC EVERYWHERE

### The Core Idea

> **Every Fabric capability â€” whether an operation, a setting, or an experience â€” should be embeddable, invocable, and composable from wherever users (and their agents) work.**

Today, "Fabric Embed" means embedding a report in a web app. That's a starting point â€” not the destination.

**Tomorrow, embedding means:**
- A developer troubleshoots a pipeline **from VS Code** â€” no portal visit needed
- A ChatGPT user provisions a workspace **in-place with a single request**
- An autonomous CI/CD agent **deploys and monitors headlessly**
- A citizen developer builds a data pipeline **inside Copilot**
- An admin configures capacity policies **from a Teams chat**, with Fabric's control plane settings surfacing as an interactive component

This is what we mean by **Fabric Everywhere**: the platform's capabilities â€” operations, configurations, and rich visual experiences alike â€” available across every surface where users and agents interact.

### Two Scenarios That Drive Everything

Within this vision, automation falls into two fundamental interaction modes:

| | **Interactive Agents** | **Autonomous Agents** |
|---|---|---|
| **Who** | Human + AI copilot, working together | Software agents operating independently |
| **How** | Conversational, visual, in-context | Headless, API-driven, unattended |
| **Surface** | MCP Apps, embedded UIs, IDE, chat | APIs, CLI, CI/CD pipelines |
| **Examples** | Troubleshooting assistant in VS Code; pipeline builder in Copilot | Nightly deployment agent; capacity scaler; automated governance |
| **Key need** | Rich, discoverable, composable experiences | Reliable, complete, well-governed APIs and tools |

Both modes depend on a shared infrastructure â€” which is why we need to invest in **three layers**.

### Two Types of Consumers That Use Fabric

In the agentic era, platforms serve two distinct consumer types:

| Consumer | Description | What They Need |
|----------|-------------|----------------|
| **ðŸ‘¤ Human Users** | Data engineers, analysts, admins who interact through portals, IDEs, chat, Teams, and the **tools they build** (scripts, CI/CD pipelines, Terraform, PowerShell) | Rich, visual, discoverable experiences wherever they work â€” plus complete, reliable APIs for the automation tools they create |
| **ðŸ¤– AI Agents** | Software agents that discover and call Fabric operations via MCP â€” a **new standalone consumer** | Discoverable tools, proper identity (SPNs), scoped permissions, audit trails |

> **Note**: Scripts, pipelines, and scheduled jobs are **tools that humans use** to extend their reach â€” not a separate persona. The same foundational infrastructure (APIs, CLI, MCP) serves both human-built automation and AI agents.

AI agents further break down into two sub-types:
- **Interactive agents**: Human + AI copilot working together in conversation (MCP Apps, Copilot in VS Code)
- **Autonomous agents**: Operate independently â€” monitoring, deploying, governing without human in the loop

> **AI agents are not just tools â€” they are standalone consumers** with their own identity, permissions, and audit trails, deserving the same first-class treatment as human users.

### Layer 1: The Foundational Layer

#### Critical Point: Build for Tomorrow's Scale

> Today's throttling limits are set for human-scale usage. But in the agentic future, **millions of agents** will interact with Fabric â€” deploying, monitoring, querying, provisioning. If we design our foundational layer for current traffic patterns, we will be caught off-guard within 2â€“3 years.

> **We need to design our infrastructure for the 100x world.**

### Layer 3: The Agentic Layer

> **Deliver intelligent, embeddable experiences â€” both interactive and autonomous â€” powered by the foundational and programmatic layers.**

This is where the value becomes visible to users and the organization. The agentic layer comprises three workstreams:

#### Workstream A: Interactive Agent Experiences (Embedding Fabric)

**The idea**: Externalize Fabric's capabilities as rich, embeddable experiences â€” not just APIs, but interactive UI components â€” that work wherever users are.

**The mechanism**: MCP Apps and equivalent technologies allow servers to serve rich, interactive UI components inline within any host (VS Code, ChatGPT, Claude, Teams, custom apps). Today we embed reports. Tomorrow we can embed:

| What We Embed | Example |
|---------------|---------|
| **Monitoring & Troubleshooting** | An MCP App that surfaces pipeline logs, identifies errors, and proposes fixes â€” rendered inline in VS Code or ChatGPT |
| **Control Plane Settings** | Capacity configuration, workspace permissions â€” interactive UIs served to any host |
| **Data Pipeline Builder** | Visual pipeline authoring experience â€” currently portal-only â€” available as an embedded component anywhere |
| **Workspace Provisioning** | AI-guided setup flow: describe your needs, see a visual preview, confirm and deploy |
| **Reports & Analytics** | Already today (Fabric Embed), expanding to more item types |

**Why this is natural for our team**: We already own Fabric Embed (currently reports/items). Expanding from embedding items to embedding *any Fabric experience* is a logical, high-impact evolution of our charter.

#### Workstream B: Autonomous Agent Access (Headless Fabric)

**The idea**: Make every Fabric operation accessible to autonomous agents â€” agents that operate without human interaction, running CI/CD, provisioning, monitoring, and governance tasks.

**Key components**:
- **Hosted MCP Server**: A unified, authenticated endpoint exposing all Fabric operations as MCP tools. External agents (GitHub Copilot, Claude, Azure AI, custom agents) connect and operate.
- **Agent Identity**: Agents are first-class personas with proper Entra ID integration, SPNs, scoped permissions, and audit trails. Every agent action goes through the same permission checks as a human user.
- **Governance Controls**: Admin controls to approve/restrict agent access, define operation scopes, enforce read-only modes.

**The promise**: *Anything you can do in the Fabric portal, an agent can do via our infrastructure â€” securely, auditably, at scale.*

> **Industry reality**: The Postman 2025 State of the API Report (5,700+ respondents) finds that while 89% of developers use AI tools, **only 24% design APIs with AI agents in mind** â€” 60% still design primarily for humans only [3]. This is the gap we must close for Fabric.

#### Workstream C: Customer MCP Tools (Fabric as an Operational Platform)

**The idea**: Let customers expose their own business logic â€” notebooks, pipelines, stored procedures â€” as MCP tools that any agent (internal or external) can discover and invoke.

**Key components**:
- **MCP Server Item**: A new Fabric item type â€” a container of custom tools hosted by Fabric
- **Publish as MCP Tool**: Wrap any Fabric item (notebook, pipeline, stored procedure) with a name, description, and parameters â†’ it becomes callable by any agent
- **Discoverability**: Custom tools appear in MCP catalogs, usable by Copilot, external agents, and other tools

**Example**: A finance team wraps their proprietary "ComputeFinanceMetric" process as an MCP tool. Now, when anyone (or any agent) needs that metric, it's a single invocation â€” no manual notebook execution required.

**Why this matters**: It transforms Fabric from an *analytics platform* into an **operational platform** â€” organizations codify their business-specific operations and make them callable by AI.

## V. MAKING IT REAL: THE TROUBLESHOOTING EXAMPLE

To show what "Embedding Fabric Everywhere" looks like in practice, let's walk through one pain point end-to-end: **Troubleshooting** â€” the #1 underserved job (opportunity score: 6.1).

### Today's Experience (The Pain)

A data engineer's pipeline fails at 2 AM:

1. Open the Fabric portal
2. Navigate to the workspace â†’ find the pipeline â†’ open run history
3. Find the failed run â†’ look at vague error message
4. Switch to a different tab â†’ open Azure Monitor
5. Search through scattered logs across Spark, Pipeline, Eventhouse
6. Manually correlate timestamps between tools
7. Google the error message â†’ find a community post from 2024
8. Try a fix â†’ redeploy â†’ wait â†’ check again

**Time to resolution: hours. Satisfaction: miserable.**

> Research by Gloria Mark at UC Irvine shows it takes an average of **23 minutes and 15 seconds** to refocus on a task after an interruption [7]. Each context switch between tools â€” portal, Azure Monitor, Spark logs, browser â€” compounds this cost. A single troubleshooting session with 5+ tool switches can waste over an hour in refocusing alone.

### Tomorrow's Experience: Foundational Layer

We invest in the infrastructure:

- **Logging API**: A unified API that aggregates logs from Azure Monitor, workspace monitoring, and Fabric-internal telemetry
- **CLI command**: `fab monitor logs --workspace <ws> --item <pipeline> --last-run` â€” one command, all relevant logs, structured output
- **Error enrichment API**: Errors come with context, suggested causes, and linked documentation

## VII. END-TO-END SCENARIOS

The three-layer architecture isn't abstract. Here are three concrete user journeys that show how the layers work together â€” each from a different persona, in a different surface, solving a real problem.

### Scenario 1: Developer Troubleshooting from VS Code

**Persona**: Sarah, Senior Data Engineer  
**Where**: VS Code with GitHub Copilot  
**Goal**: Diagnose and fix a failing Spark notebook without opening the Fabric portal

1. Sarah's CI/CD pipeline fails with a vague Spark error. She opens VS Code and asks Copilot: *"Why did my Spark notebook fail in the Production workspace?"*
2. Copilot invokes the **Fabric MCP tools** to pull the latest run logs and error details from the Fabric workspace.
3. The agent analyzes the logs and identifies a memory configuration mismatch â€” the notebook's Spark pool doesn't have enough executor memory for the new data volume.
4. Copilot suggests a fix: update the Spark pool configuration. Sarah approves.
5. The agent executes the change via the **CLI** (or API) â€” updating the pool settings without Sarah ever leaving VS Code.
6. Copilot re-triggers the notebook run through the **API** and monitors the result.
7. The run succeeds. Sarah commits the updated configuration to Git and moves on â€” **total time: 4 minutes** vs. 20+ minutes of portal navigation.

**Layers used**: Foundational (APIs, CLI, logs) â†’ Programmatic (CI/CD trigger, scriptable config) â†’ Agentic (Copilot-driven interactive troubleshooting)

### Scenario 2: Business User Creating a Report from Claude Desktop

**Persona**: David, Marketing Analytics Manager  
**Where**: Claude Desktop with Fabric MCP  
**Goal**: Build a campaign performance report without learning the Fabric portal

1. David opens Claude Desktop and says: *"I need a report showing our Q4 marketing campaign performance â€” impressions, conversions, and ROI by channel."*
2. Claude invokes the **Fabric MCP** to discover available semantic models and lakehouses in David's workspace.
3. The agent finds the `Marketing_Analytics` semantic model and lists the relevant measures and dimensions.
4. Claude suggests a report layout: a bar chart for impressions by channel, a conversion funnel, and an ROI table. David says *"Add a trend line for week-over-week."*
5. The agent creates the report via the **API**, applying David's customizations.
6. Claude returns a link to the live report in Fabric â€” David can share it directly with his team.
7. David asks *"Schedule this to refresh every Monday morning"* â€” the agent sets up the refresh schedule via the **API**.

**Layers used**: Foundational (APIs, semantic model access) â†’ Programmatic (automated refresh scheduling) â†’ Agentic (Claude-driven interactive creation)

### Scenario 3: Autonomous CI/CD Agent in GitHub Actions

**Persona**: None (fully autonomous)  
**Where**: GitHub Actions  
**Goal**: Deploy workspace changes, run validation, and roll back if tests fail â€” with zero human intervention

1. A developer merges a PR that updates a Spark notebook and a pipeline definition.
2. GitHub Actions triggers the deployment workflow. The **CI/CD agent** authenticates to Fabric using a Service Principal.
3. The agent calls the **Fabric API** to deploy the updated items to the Staging workspace.
4. The agent invokes the **CLI** to run the validation pipeline: `fab pipeline run --workspace Staging --pipeline ValidationSuite`.
5. The validation pipeline completes with 2 test failures. The agent pulls the test results via the **API**.
6. Based on the failure policy (configured in the workflow YAML), the agent automatically rolls back the Staging workspace to the previous version.
7. The agent posts a summary to the PR: *"Deployment rolled back â€” 2 validation failures detected. See attached test report."* The developer reviews and fixes.

**Layers used**: Foundational (APIs, SPN identity, CLI) â†’ Programmatic (CI/CD pipeline, scriptable deployment & rollback) â†’ Agentic (autonomous decision-making, zero human intervention)

### Why These Scenarios Matter

Each scenario demonstrates a different point on the automation spectrum:

| Scenario | Automation Level | Human Involvement | Key Enablers |
|----------|-----------------|-------------------|--------------|
| **Developer in VS Code** | Interactive (human-in-the-loop) | Approves each step | MCP tools, CLI, APIs |
| **Business User in Claude** | Guided (AI suggests, human refines) | Provides intent, reviews output | MCP discovery, APIs, semantic models |
| **CI/CD Agent in GitHub** | Autonomous (zero-touch) | Reviews after the fact | SPN identity, APIs, CLI, governance policies |

> **The key insight**: The same three layers power all three scenarios. The difference is the level of human involvement â€” which is configurable, not hard-coded.
>
> **Bottom line**: Fabric powers the outcome, even when users never open the portal.
