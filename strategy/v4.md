## PART V: OUR DUAL STRATEGIC DIRECTIONS

### 5.1 The Strategic Framework: Hybrid Convergence

Alon Baram's guidance (Dec 24, 2025):
> "You are not bound to any paradigm, though **a combination of tools is more likely to be accepted** (e.g. CLI, MCP and a CI/CD tool like Terraform or Git, etc.)"

**Our Approach**: Not "either/or" but "both, with smart phasing."

### 5.2 Direction 1: Bring Professional Developers INTO Fabric

**Goal**: Make Fabric a first-class platform for professional developers through native, secure, and scalable automation experiences.

**How We Do This**:
- Provide complete CLI and Terraform experience for provisioning, managing, and deploying Fabric resources
- Ensure every Fabric workload is **API-first** and **SPN-ready**
- Deliver **Hosted MCP** (we own thisã€meeting contextã€‘) for AI agents and Copilots to orchestrate Fabric operations securely
- Support CI/CD workflows through backend APIs and SDKs that power Git integration, deployment pipelines, and variables
- Introduce **Fabric Sandbox** (ephemeral workspace) for safe experimentation and iteration
- Enable **Fabric Cloud Shell** (browser-based CLI, pre-authenticated) for in-portal automation

**Example Scenario**: A platform engineer at a large enterprise wants to automate provisioning of 100 Fabric workspaces with preconfigured pipelines and access policies. Using our Terraform provider and CLI, they can do this in **minutes** â€” without touching the UI.

**Impact**: Directly addresses CAT Research's top 4 underserved jobs (CI/CD, monitoring, capacity planning, access management).

### 5.3 Direction 2: Bring Fabric TO Professional Developers (Where They Are)

**Goal**: Meet developers in their native environments by integrating Fabric deeply into VS Code, GitHub, and local development workflows.

**How We Do This**:
- Integrate Fabric CLI and **Local MCP** (we own thisã€meeting contextã€‘) into VS Code, GitHub Copilot, and Claude
- Enable local iteration and validation through **Local MCP** and **CLI Script Items**
- Support developer-first workflows like "git push â†’ deploy" using familiar tools
- Provide **sandboxed CLI sessions in VS Code** for safe experimentation without full tenant permissions
- Enable **Blueprint-to-code** conversion (AI-generated specs â†’ Terraform/CLI scripts)

**Example Scenario**: A data engineer working in VS Code wants to test a Fabric pipeline locally before pushing to production. With **Local MCP** and CLI integration, they can validate the pipeline, commit to Git, and trigger deployment â€” all from their IDE.

**Impact**: Aligns with Kim Manis's vision of "a productive, familiar IDE-like environment with a great multitasking experience"[17](https://microsoft.sharepoint.com/teams/DataCloud/_layouts/15/Doc.aspx?sourcedoc=%7BC14DAF5E-9B72-43FF-97A3-BDF146BF219F%7D&file=Draft%20Kr%20Plan.pptx&action=edit&mobileredirect=true&DefaultItemOpen=1)and reduces context switching (costs ~23 minutes per interruption[6](https://microsofteur-my.sharepoint.com/personal/habos_microsoft_com/_layouts/15/Doc.aspx?sourcedoc=%7BA3FA9806-3E6B-42EC-8480-BB0D53F8D4AE%7D&file=[SPEC]%20Fabric%20Local%20MCP.docx&action=default&mobileredirect=true&DefaultItemOpen=1)).

### 5.4 Why Both Directions Matter

**In-Product (Direction 1)** enables:
- Lower friction onboarding (pre-authenticated, no local setup)
- Tight Copilot integration (unified experience)
- Easier governance (enterprise control)
- Long-tail automation scenarios (too complex for Copilot alone, too lightweight for full CI/CD)

**Out-of-Product (Direction 2)** enables:
- Meets developers in #1 IDE (VS Code)
- NPS measurement via in-tool surveys (solves API NPS problem)
- Local development, offline scenarios, Git workflows
- Blueprint-to-code conversion (AI-generated specs â†’ version-controlled automation)
